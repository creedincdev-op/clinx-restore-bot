Feature Page // Runtime Safety

CLINX turns raw admin power into controlled recovery execution.

The Safety Gate adds runtime command tiers on top of Discord permissions. Guild owners stay fully trusted, owners can grant CLINX-trusted admins, and untrusted admins can still request powerful operations without being able to execute them alone.

Safety tiers compressed into one operating model

Public commands stay open, safe admin actions stay direct, and risky writes route through trust checks or owner approval before CLINX touches the live guild.

Trust model in plain English

The actual guild owner is the root of trust. CLINX stores trusted admins per guild in a separate safety file and evaluates every protected command at runtime.

Owner

Always trusted. Can run every tier, including governance and `/leave`.

  • Bypasses all runtime gates
  • Can grant or revoke trusted admins
  • Is the only approver for protected requests

Trusted Admin

Fast lane for serious operators who still should not own governance commands.

  • Runs Tier 2 and Tier 3 directly
  • Cannot bypass Tier 4
  • Does not need owner approval for protected recovery work

Untrusted Admin

Can still work, but high-impact actions become requests instead of direct execution.

  • Runs Tier 1 directly
  • Tier 2 and Tier 3 create approval cards
  • Cannot execute `/leave` or trust management

Owner approval flow stays visible and frozen

When an untrusted admin requests a protected action, CLINX does not run it. It creates a public approval card in the same channel, tags the owner, and locks the payload before anything can execute.

01

Request created

Requester, command route, backup ID, selected lanes, and planned changes are captured immediately.

02

Owner is tagged

Only the actual guild owner can use the Approve or Deny buttons on the public request card.

03

Timed decision

Requests expire after 2 minutes, and each guild can only have one pending protected request at a time.

04

Frozen execution

The approved run uses the exact frozen payload from the request and updates the same card to started, completed, failed, denied, or expired.

Operating rules that stay tight under pressure

One pending protected request per guild Stops approval spam and prevents owners from approving the wrong payload.
Global throttle path stays centralized Protected bulk operations still go through the same paced dispatch gate and Retry-After handling.
Backup ownership stays unchanged Users still only see and use their own backup IDs while safety gating wraps the write path around them.
Status cards stay honest Approved runs update the same card to started, completed, failed, denied, or expired so the channel tells the truth.
